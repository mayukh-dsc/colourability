## What This Project Is

**colourability** is a **zero-dependency, open-source TypeScript module** that applies color-blindness filters, grayscale, and optional stylized matrices to any webpage via a single uniform mechanism.

It is designed to:
- Be plugged into **any website** via `<script>` tag or npm import
- Work across **mobile, desktop, and large-format screens** (TVs, kiosks)
- Correctly filter **HTML video, `<canvas>`, WebGL, and static DOM content**
- Be **tiny** (< 4 KB minified + gzipped), with no runtime dependencies

See `PLAN.md` for the full phased development plan.

## Architecture

### Single-Mechanism: SVG `<feColorMatrix>`

Every filter — including grayscale — is expressed as a 4×5 `feColorMatrix` injected into `<head>` and applied to `document.documentElement` via CSS `filter: url(#id)`.

```typescript
// 1. Inject once into <head>
// <svg id="__colourability-svg__" style="display:none">
//   <defs>
//     <filter id="__colourability-active__" color-interpolation-filters="linearRGB">
//       <feColorMatrix type="matrix" values="..." />
//     </filter>
//   </defs>
// </svg>

// 2. Point the root element at it
document.documentElement.style.filter = 'url(#__colourability-active__)';
```

**Why a single mechanism?**
- Grayscale, color-blindness simulation, daltonization, and stylized matrices are all 4×5 linear matrix operations — they fit `feColorMatrix` exactly
- No overlay div, no CSS filter shorthand strings, no blend modes — no branching
- `color-interpolation-filters="linearRGB"` on every `<filter>` ensures gamma-correct output, which is critical for color-blindness accuracy

**Why it works on video and canvas:**
Applying `filter: url(...)` to `document.documentElement` composites at the GPU/paint level, the same way `filter: grayscale()` does. It affects all content including `<video>`, `<canvas>`, and WebGL.

### File Responsibilities

| File | Role |
|------|------|
| `src/index.ts` | `Colourability` class, public API surface |
| `src/types.ts` | `FilterOptions`, `FilterState` |
| `src/matrix.ts` | `IDENTITY_MATRIX`, `parseMatrixValues`, `interpolateMatrix`; re-exports `FILTERS` / `GRAYSCALE` for tests and modules that import utilities alongside the registry |
| `src/filters/*.ts` | Built-in matrix strings by category; `src/filters/index.ts` aggregates `FILTERS` |
| `src/overlay.ts` | DOM lifecycle: inject SVG, update matrix, remove SVG, restore styles |

Built-in preset ids are **namespaced** strings (`category/subname`). They are registered in `src/filters/index.ts`.

## Built-in Filters

| Name | Type | Description |
|------|------|-------------|
| `visual/grayscale` | Visual | BT.709 luminance desaturation |
| `simulateColorBlindness/protanopia` | CB Simulation | Red-cone (L) deficiency |
| `simulateColorBlindness/deuteranopia` | CB Simulation | Green-cone (M) deficiency |
| `simulateColorBlindness/tritanopia` | CB Simulation | Blue-cone (S) deficiency (Machado-style matrix) |
| `daltonizeColorBlindness/protanopia` | CB Compensation | Linear compensation heuristic (`2I − M_sim`) |
| `daltonizeColorBlindness/deuteranopia` | CB Compensation | Same |
| `daltonizeColorBlindness/tritanopia` | CB Compensation | Same |
| `fun/sepia` | Stylized | Sepia-style RGB mix |
| `fun/invert` | Stylized | RGB invert |

All built-ins support `setIntensity(0–1)` via `interpolateMatrix()` in `src/matrix.ts`. Custom matrices from `applyMatrix()` use the same interpolation.

## Public API Contract

The API must remain stable. Do not change method signatures without updating `README.md` and `CHANGELOG.md`. Types live in `src/types.ts` and are re-exported from `src/index.ts` — `index.d.ts` is generated by `tsc`, do not hand-edit it.

```typescript
const colourability = new Colourability();

colourability.apply(name: string, options?: FilterOptions): this
colourability.applyMatrix(matrix: string, options?: FilterOptions): this
colourability.remove(): this
colourability.setIntensity(value: number): this   // 0–1
colourability.list(): string[]
colourability.getState(): FilterState
```

**Deferred to v2 (not in v1):** `register()`, `on()` / events, `safeMode` — see `PLAN.md`.

All public types live in `src/types.ts`:

```typescript
export interface FilterOptions {
  intensity?: number;   // 0–1, default 1
}

export interface FilterState {
  active: string | null;       // namespaced preset id, or null when custom / none
  customMatrix: string | null; // base matrix when applyMatrix is active
  options: FilterOptions;
}
```

Preset lookup is `FILTERS[name]` in `src/filters/index.ts` — type `Record<string, string>` (20-value `feColorMatrix` row-major strings).

## Coding Conventions

### Language: TypeScript (strict)
- **All source files are `.ts`** — no `.js` files in `src/`
- `tsconfig.json` must have `"strict": true`, `"noUncheckedIndexedAccess": true`, `"exactOptionalPropertyTypes": true`
- **Never use `any`** — use `unknown` and narrow, or define a proper type in `src/types.ts`
- **No type assertions (`as`)** unless accompanied by a comment explaining why it is safe
- All exported symbols must have explicit return types — do not rely on inference for public API
- **No classes in `src/overlay.ts`** — use factory functions; `Colourability` in `src/index.ts` is the only class

### General Style
- ES2020+ syntax: optional chaining, nullish coalescing, `const`/`let`, never `var`
- No runtime dependencies — devDependencies only
- Keep `src/` imports self-contained; never import from `test/` or config files

### Documentation
- **Always keep `README.md` updated** when the public API or filter list changes
- Public API types are the documentation — do not duplicate with JSDoc unless the intent is non-obvious
- Non-obvious logic must have an inline comment explaining *why*, not *what*

### DOM Safety Rules (Critical)
1. **Always restore `document.documentElement.style.filter` on `remove()`** — store the previous value before overwriting; restore it exactly on teardown
2. **Use a single shared `<svg id="__colourability-svg__">` in `<head>`** — do not inject multiple `<svg>` elements; update the `<feColorMatrix values>` attribute in place when switching filters
3. **Always set `color-interpolation-filters="linearRGB"`** on every `<filter>` element — gamma correctness is not optional; omitting it silently breaks all color-blindness filters
4. **Remove the `<svg>` from `<head>` on `remove()`** — do not leave orphaned SVG nodes

### Matrix utilities (`src/matrix.ts`) and presets (`src/filters/`)

- `src/matrix.ts` holds **shared utilities**: `IDENTITY_MATRIX`, `parseMatrixValues`, `interpolateMatrix`, and re-exports `FILTERS` / `GRAYSCALE` for tests and tree-shaking consumers of types.
- **Built-in matrices** live under `src/filters/` grouped by category. The aggregator `src/filters/index.ts` exports `FILTERS: Record<string, string>`.

Do not inline ad-hoc 20-value matrix strings in `src/index.ts` or `overlay.ts` — add a named preset or pass through `applyMatrix` at the call site.

### Testing (Required)
- **Every function and method must have unit tests** — do not ship untested code
- Tests live in `src/__tests__/` co-located with source, named `*.test.ts`
- Use **Vitest** with `@vitest/coverage-v8`
- **Write the test first (TDD)** when implementing new features
- Test categories per file:
  - `matrix.test.ts` — `FILTERS` keys are valid namespaced ids with 20-coefficient strings; `interpolateMatrix()` at t=0 → identity, t=1 → original
  - `overlay.test.ts` — SVG injection with `happy-dom`: idempotency, `linearRGB` attribute present, teardown restores styles
  - `index.test.ts` — public API behaviour and integration across modules
- Minimum coverage targets: **90% statements, 85% branches**

### Bundling
- Bundler: **Rollup** with `@rollup/plugin-typescript` and `@rollup/plugin-terser`
- Three output formats: `umd`, `esm`, `iife`
- `index.d.ts` is **generated by `tsc`** (`declaration: true`) — never hand-edit it
- Do not import bundler config or devDependencies inside `src/`
- **Entry bundle exports only the default `Colourability` class** so IIFE/UMD globals stay `new Colourability()` without a `.default` wrapper

## Known Limitations (Do Not Try to "Fix" Without Discussion)

| Limitation | Reason to leave it |
|---|---|
| `filter: url(...)` on `<html>` creates a new stacking context | Browser constraint — `position:fixed` on the host page may behave differently. A mitigation (e.g. skip root filter) is **v2** scope — do not add a half-baked `safeMode` to v1. |
| Cross-origin iframes unaffected | Browser security boundary — intentional |
| Daltonization matrices are pre-computed linear approximations | Full accuracy requires runtime LMS conversion, which is per-pixel and too slow for a page-level filter. The pre-computed matrix with `linearRGB` interpolation is the industry standard used by Chrome DevTools and OS accessibility tools. Do not replace with a canvas pixel loop. |

## What NOT to Do

- Do not use `document.body.style.filter` as the filter target — use `document.documentElement`
- Do not add runtime dependencies to `package.json` (devDependencies are fine)
- Do not manipulate `<video>` or `<canvas>` pixels directly (no `getImageData` / WebGL shader injection)
- Do not use `any` — ever
- Do not hand-edit `index.d.ts` — it is generated by `tsc`
- Do not write `.js` files in `src/` — TypeScript only
- Do not skip writing tests for a new function or method, even if it seems trivial
- Do not omit `color-interpolation-filters="linearRGB"` from SVG `<filter>` elements
- Do not inject multiple `<svg>` elements into `<head>` — update the single shared one in place
- Do not attempt per-pixel daltonization via `canvas.getImageData()` — use the SVG matrix approach
- Do not add non-matrix filters (overlay div, blend modes, CSS shorthand strings) — all filters must be expressible as a `feColorMatrix`
