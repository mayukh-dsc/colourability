## What This Project Is

`overlay-filter` is a **zero-dependency, open-source TypeScript module** that applies color-blindness filters and a grayscale filter to any webpage via a single uniform mechanism.

It is designed to:
- Be plugged into **any website** via `<script>` tag or npm import
- Work across **mobile, desktop, and large-format screens** (TVs, kiosks)
- Correctly filter **HTML video, `<canvas>`, WebGL, and static DOM content**
- Be **tiny** (< 4 KB minified + gzipped), with no runtime dependencies

See `PLAN.md` for the full phased development plan.

## Architecture

### Single-Mechanism: SVG `<feColorMatrix>`

Every filter — including grayscale — is expressed as a 4×5 `feColorMatrix` injected into `<head>` and applied to `document.documentElement` via CSS `filter: url(#id)`.

```typescript
// 1. Inject once into <head>
// <svg id="__overlay-filter-svg__" style="display:none">
//   <defs>
//     <filter id="__of-active__" color-interpolation-filters="linearRGB">
//       <feColorMatrix type="matrix" values="..." />
//     </filter>
//   </defs>
// </svg>

// 2. Point the root element at it
document.documentElement.style.filter = 'url(#__of-active__)';
```

**Why a single mechanism?**
- Grayscale, color-blindness simulation, and daltonization are all 4×5 linear matrix operations — they fit `feColorMatrix` exactly
- No overlay div, no CSS filter shorthand strings, no blend modes — no branching
- `color-interpolation-filters="linearRGB"` on every `<filter>` ensures gamma-correct output, which is critical for color-blindness accuracy

**Why it works on video and canvas:**
Applying `filter: url(...)` to `document.documentElement` composites at the GPU/paint level, the same way `filter: grayscale()` does. It affects all content including `<video>`, `<canvas>`, and WebGL.

### File Responsibilities

| File | Role |
|------|------|
| `src/index.ts` | `OverlayFilter` class, public API surface |
| `src/types.ts` | `FilterOptions`, `FilterState` only |
| `src/matrix.ts` | Matrix constants, `FILTERS` (`Record<string, string>`), `interpolateMatrix()` |
| `src/overlay.ts` | DOM lifecycle: inject SVG, update matrix, remove SVG, restore styles |

There is **no** `src/filters.ts` — preset names map directly to matrix strings in `src/matrix.ts`.

## Built-in Filters

| Name | Type | Description |
|------|------|-------------|
| `grayscale` | Visual | Desaturates the entire page |
| `simulate-protanopia` | CB Simulation | Shows what a red-cone deficient person sees |
| `simulate-deuteranopia` | CB Simulation | Shows what a green-cone deficient person sees |
| `simulate-tritanopia` | CB Simulation | Shows what a blue-cone deficient person sees |
| `daltonize-protanopia` | CB Compensation | Shifts colors to improve visibility for red-cone deficiency |
| `daltonize-deuteranopia` | CB Compensation | Shifts colors to improve visibility for green-cone deficiency |
| `daltonize-tritanopia` | CB Compensation | Shifts colors to improve visibility for blue-cone deficiency |

All filters support `setIntensity(0–1)` via matrix interpolation (`interpolateMatrix()` in `src/matrix.ts`).

## Public API Contract

The API must remain stable. Do not change method signatures without updating `README.md`. Types live in `src/types.ts` and are re-exported from `src/index.ts` — `index.d.ts` is generated by `tsc`, do not hand-edit it.

```typescript
const filter = new OverlayFilter();

filter.apply(name: string, options?: FilterOptions): this
filter.remove(): this
filter.setIntensity(value: number): this   // 0–1
filter.list(): string[]
filter.getState(): FilterState
```

**Deferred to v2 (not in v1):** `register()`, `on()` / events, `safeMode` — see `PLAN.md`.

All public types live in `src/types.ts`:

```typescript
// src/types.ts

export interface FilterOptions {
  intensity?: number;   // 0–1, default 1
}

export interface FilterState {
  active: string | null;
  options: FilterOptions;
}
```

Preset lookup is `FILTERS[name]` in `src/matrix.ts` — type `Record<string, string>` (20-value `feColorMatrix` row-major strings).

## Coding Conventions

### Language: TypeScript (strict)
- **All source files are `.ts`** — no `.js` files in `src/`
- `tsconfig.json` must have `"strict": true`, `"noUncheckedIndexedAccess": true`, `"exactOptionalPropertyTypes": true`
- **Never use `any`** — use `unknown` and narrow, or define a proper type in `src/types.ts`
- **No type assertions (`as`)** unless accompanied by a comment explaining why it is safe
- All exported symbols must have explicit return types — do not rely on inference for public API
- **No classes in `src/overlay.ts`** — use factory functions; `OverlayFilter` in `src/index.ts` is the only class

### General Style
- ES2020+ syntax: optional chaining, nullish coalescing, `const`/`let`, never `var`
- No runtime dependencies — devDependencies only
- Keep `src/` imports self-contained; never import from `test/` or config files

### Documentation
- **Always keep `README.md` updated** when the public API or filter list changes
- Public API types are the documentation — do not duplicate with JSDoc unless the intent is non-obvious
- Non-obvious logic must have an inline comment explaining *why*, not *what*

### DOM Safety Rules (Critical)
1. **Always restore `document.documentElement.style.filter` on `remove()`** — store the previous value before overwriting; restore it exactly on teardown
2. **Use a single shared `<svg id="__overlay-filter-svg__">` in `<head>`** — do not inject multiple `<svg>` elements; update the `<feColorMatrix values>` attribute in place when switching filters
3. **Always set `color-interpolation-filters="linearRGB"`** on every `<filter>` element — gamma correctness is not optional; omitting it silently breaks all color-blindness filters
4. **Remove the `<svg>` from `<head>` on `remove()`** — do not leave orphaned SVG nodes

## Known Limitations (Do Not Try to "Fix" Without Discussion)

| Limitation | Reason to leave it |
|---|---|
| `filter: url(...)` on `<html>` creates a new stacking context | Browser constraint — `position:fixed` on the host page may behave differently. A mitigation (e.g. skip root filter) is **v2** scope — do not add a half-baked `safeMode` to v1. |
| Cross-origin iframes unaffected | Browser security boundary — intentional |
| Daltonization matrices are pre-computed linear approximations | Full accuracy requires runtime LMS conversion, which is per-pixel and too slow for a page-level filter. The pre-computed matrix with `linearRGB` interpolation is the industry standard used by Chrome DevTools and OS accessibility tools. Do not replace with a canvas pixel loop. |

## What NOT to Do

- Do not use `document.body.style.filter` as the filter target — use `document.documentElement`
- Do not add runtime dependencies to `package.json` (devDependencies are fine)
- Do not manipulate `<video>` or `<canvas>` pixels directly (no `getImageData` / WebGL shader injection)
- Do not use `any` — ever
- Do not hand-edit `index.d.ts` — it is generated by `tsc`
- Do not write `.js` files in `src/` — TypeScript only
- Do not skip writing tests for a new function or method, even if it seems trivial
- Do not add `src/filters.ts` — presets live in `src/matrix.ts` as `FILTERS`
- Do not omit `color-interpolation-filters="linearRGB"` from SVG `<filter>` elements
- Do not inject multiple `<svg>` elements into `<head>` — update the single shared one in place
- Do not attempt per-pixel daltonization via `canvas.getImageData()` — use the SVG matrix approach
- Do not add non-matrix filters (overlay div, blend modes, CSS shorthand strings) — all filters must be expressible as a `feColorMatrix`
